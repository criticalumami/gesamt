# Meticulous Explanation of the "Gesamtkunstwerk" Project

This document provides a detailed breakdown of the "Gesamtkunstwerk" project, covering its architecture, backend, frontend, and deployment process.

### 1. High-Level Purpose & Architecture

**Gesamtkunstwerk** (a German word meaning "total work of art") is a sophisticated, automated job searching and tracking application. Its primary purpose is to:
1.  **Scrape** job listings from multiple online platforms based on user-defined keywords and locations.
2.  **Analyze** these listings using AI (Google's Gemini) to score their relevance against a user's profile.
3.  **Present** the matched jobs in a clean, minimalist web interface.
4.  Provide tools for **tracking** the application status for each job (Applied, Interviewing, etc.).
5.  Include a secondary dashboard for **managing outreach** to a custom list of companies or offices.

The application follows a modern **decoupled architecture**:
*   A **Python backend** built with the **FastAPI** framework. This backend is responsible for all the heavy lifting: serving the web pages, running the scrapers, interacting with the database, and providing a REST API for the frontend.
*   A **vanilla JavaScript frontend** that runs entirely in the user's browser. It communicates with the backend via API calls to fetch data and trigger actions. This makes the UI fast and responsive without needing page reloads.
*   A **SQLite database** (`jobs.db`) for persistent storage of job listings, application statuses, and user settings. SQLite is a lightweight, file-based database that is simple to manage.

### 2. Backend Deep Dive

The backend is the engine of the application, composed of three main Python files: `db.py`, `scraper.py`, and `server.py`.

#### a. Database (`db.py`)

This file manages all interactions with the SQLite database.

*   **Schema:** It defines two main tables:
    1.  `jobs`: Stores the scraped job listings. Key columns include `url` (the primary key), `platform`, `title`, `description`, `status` (e.g., 'New', 'Applied', 'Expired'), and the AI-generated fields: `match_score`, `match_reason`, and `key_requirements`.
    2.  `office_applications`: Stores the status and notes for the custom outreach list found in the "Custom Directory" tab.
*   **Key Functions:**
    *   `init_db()`: Creates the database tables if they don't exist.
    *   `save_job()`: Saves a single job into the `jobs` table. It uses `INSERT OR REPLACE` to avoid duplicates based on the job URL.
    *   `get_all_jobs_df()`: Retrieves all non-expired jobs from the database and returns them as a pandas DataFrame, which is convenient for data manipulation in the backend.
    *   `update_job_status()`: Updates the `status` of a job in the database (e.g., when you change a job from 'New' to 'Applied').
    *   `prepare_new_scan()`: A maintenance function that runs before a new scrape. It marks old, un-interacted-with jobs as 'Expired' to keep the main feed clean.

#### b. Web Scraper (`scraper.py`)

This is the most complex part of the backend. It's responsible for fetching job data from the web.

*   **Multi-Platform Support:** It contains dedicated functions to scrape various job boards, including:
    *   `scrape_daleel_madani()`
    *   `scrape_un_careers()`
    *   `scrape_linkedin()`
    *   `scrape_reliefweb()`
    *   And several others.
*   **Scraping Techniques:** It employs a variety of techniques to handle different websites:
    *   **HTTP Requests:** For simple, server-rendered websites (like ReliefWeb), it uses libraries like `requests` and `curl_cffi` to download the HTML directly.
    *   **Browser Automation:** For complex, JavaScript-heavy websites (like the UN Careers portal), it uses the **Playwright** library to control a real web browser, log in, click buttons, and extract data just as a human would.
    *   **Anti-Blocking Measures:** The code shows significant effort to avoid being blocked, using techniques like rotating user agents, handling cookies, adding delays, and using `curl-cffi` to impersonate real browser fingerprints.
*   **AI Integration (Gemini):**
    *   `get_ai_evaluation()`: After scraping the text of a job description, this function sends the text to the Gemini API. It asks the AI to score the job's relevance based on your configured profile/keywords, provide a reason for the score, and extract key requirements. This is the core of the "profile-matching" feature.
    *   `expand_keywords_with_ai()`: An auxiliary function that uses your resume/profile to ask Gemini to suggest additional relevant keywords for your job search.
*   **Execution Flow (`run_pipeline()`):** This main function orchestrates the entire scraping process. It runs the individual scrapers for the selected platforms in parallel using Python's `threading` module to speed things up.

#### c. API Server (`server.py`)

This file ties everything together using the **FastAPI** framework. It defines the API endpoints that the frontend communicates with.

*   **Serving the Frontend:**
    *   `@app.get("/")`: The root endpoint serves the main `index.html` file.
    *   `app.mount("/", StaticFiles(directory=static_dir))`: This command tells FastAPI to serve all the files from the `static` directory (like `app.js`, `style.css`, `favicon.svg`) directly.
*   **Core API Endpoints:**
    *   `GET /api/jobs`: Fetches all current job listings from the database via `db.py`.
    *   `GET /api/settings`: Retrieves the user's saved settings (keywords, locations, etc.) from `settings.json`.
    *   `POST /api/settings`: Saves the user's settings to `settings.json`.
    *   `POST /api/scan`: This is a crucial endpoint. When called, it starts the entire scraping pipeline (`scraper.run_pipeline()`) in a background process.
    *   `GET /api/logs`: Allows the frontend to poll for the live log output of the running scraper process, which is displayed in the terminal window in the UI.
    *   `GET /api/analytics`: Calculates and returns data for the analytics dashboard (KPIs, funnel data, etc.).
    *   `POST /api/resume/upload`: Handles the resume upload, extracts the text, and uses the Gemini API to automatically generate a profile summary and keywords.

### 3. Frontend Deep Dive

The frontend is what you see and interact with in your browser. It's built with standard HTML, CSS, and JavaScript, without a complex framework like React or Vue, which keeps it lightweight.

#### a. HTML Structure (`static/index.html`)

The HTML file defines the skeleton of the user interface.

*   **Header:** Contains the title "GESAMTKUNSTWERK ☭" and the main tab navigation buttons (Global Feed, Custom Directory, etc.).
*   **Tab Panels:** The main content is divided into `<section>` elements that act as panels for each tab. Only the active tab's panel is visible at one time.
*   **Key Sections:**
    *   `panel-global`: The main job feed, including the search bar, action buttons, the terminal output console, and the grid where job cards are displayed.
    *   `panel-custom`: The directory for custom outreach.
    *   `panel-analytics`: The container for the analytics charts.
    *   `panel-settings`: The form for configuring keywords, locations, API keys, etc.
*   **Detail Drawer:** A slide-out panel (`<div class="detail-drawer">`) used to show the full details of a selected job without leaving the main page.
*   **Scripts & Styles:** It links to the `style.css` stylesheet and the main `app.js` JavaScript file at the end of the body.

#### b. CSS Styling (`static/style.css`)

This file defines the application's visual appearance, which the `README.md` describes as a "Minimalist Swiss Grid" style.

*   **Design System:** It uses CSS variables (e.g., `--bg`, `--text`, `--border`) to create a consistent and easily themeable (light/dark) design. The style is characterized by:
    *   **Minimalism:** A grayscale color palette.
    *   **Sharp Geometry:** No rounded corners (`border-radius: 0px !important`).
    *   **Strong Typography:** Use of the "Inter" font, bold weights, and uppercase text for headings.
    *   **Grid Layout:** The main content areas use CSS Grid for alignment.
*   **Key Styles:**
    *   `.glass-header`: The fixed navigation bar at the top.
    *   `.card`: The style for the individual job listing cards.
    *   `.detail-drawer`: The styles for the slide-out panel, including its animation.
    *   `.chart-container`: The recently added style to constrain the height of the analytics charts.

#### c. JavaScript Logic (`static/app.js`)

This is the "brain" of the frontend. It controls everything the user sees and does.

*   **State Management:** A global `state` object at the top holds all the application's data (the list of jobs, settings, the current search query, etc.). When data is fetched from the backend, it's stored here.
*   **Initialization:** When the page loads (`DOMContentLoaded`), the script runs three main functions:
    1.  `initTabs()`: Sets up the click listeners for the main navigation tabs to switch between panels.
    2.  `initEventListeners()`: Sets up all other event listeners, like for the search inputs and buttons.
    3.  `loadAllData()`: Makes the initial API calls to `/api/settings`, `/api/jobs`, and `/api/offices` to populate the UI with data when the page first loads.
*   **Rendering Functions:** A set of `render...()` functions are responsible for generating HTML and updating the UI based on the current `state`.
    *   `renderJobs()`: Loops through the `state.jobs` array and creates the HTML for each job card, then injects it into the listings grid. This is where the UI is dynamically built.
    *   `renderAnalytics()`: Generates the HTML for the analytics dashboard and uses the **Chart.js** library to create the charts.
    *   `renderOffices()`: Renders the cards for the custom directory.
*   **API Actions:** Contains `async` functions that use the `fetch` API to communicate with the backend. Every time you click "Save Settings", "Scan Job Boards", or update a job's status, one of these functions is called to send a request to the corresponding endpoint on the `server.py` backend.

### 4. Deployment & Workflow

*   **Dockerfile:** This file contains instructions to package the entire application (both the Python backend and the static frontend files) into a **Docker container**. This is a standard way to ensure that the application runs consistently across different environments.
*   **GitHub Actions (`.github/workflows/deploy.yml`):** This file defines the **Continuous Deployment (CD)** pipeline.
    *   **Trigger:** The workflow is triggered automatically on every `push` to the `main` or `lean-rewrite` branches.
    *   **Action:** It checks out the code and then executes a `git push --force` to a special Git repository on **Hugging Face Spaces**. Hugging Face then automatically builds the Docker container from the `Dockerfile` and deploys the application, making it live on the web.

This automated workflow means that to deploy a new version of the application, all one has to do is push the code to the correct GitHub branch.
